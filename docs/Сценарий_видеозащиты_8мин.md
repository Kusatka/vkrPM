# Вотчер — сценарий видео-защиты (до 8 минут)

## Главная мысль (чтобы не утонуть в стеке)

Защищаем **понятный Django-сервис**: парсинг расписаний → каталог фильмов → история цен → сравнение по кинотеатрам и график. Celery, Telegram, Docker, PostgreSQL называем **одной фразой** как «дополнительные модули / план развития» — не делаем их центром рассказа.

## Подготовка к записи (один раз)

В активированном venv приведи базу к ОДНОМУ состоянию (не смешивать):

- **Вариант А — демо, стабильно, без интернета** (проще для записи):
  `python manage.py flush` → `python manage.py seed_demo`
  4 фильма с историей цен и графиком; демо-строки честно помечены бейджем «демо».
- **Вариант Б — живые данные**:
  `python manage.py flush` → `python manage.py scrape` (нужен интернет)
  Реальные фильмы и кнопки «Билеты». График истории появится, когда накопится 2+ прогона.

Затем: `python manage.py runserver`, открой заранее вкладки: `/` (афиша), `/new/` (новинки), страница любого фильма, `/admin/scraping/scraperun/` (журнал).
Запись 1920×1080, экран + микрофон (OBS или Win+G).

## Сценарий по минутам

**0:00–0:45 — Проблема.** «Цены и расписания кино разбросаны по сайтам кинотеатров, историю цен никто не хранит, а показы в оригинале теряются в общем списке. Вотчер собирает это в одном месте и показывает, где и когда дешевле».

**0:45–1:30 — Что это и стек (коротко).** «Это Django-сервис. Парсеры собирают сеансы с сайтов, данные ложатся в каталог фильмов и кинотеатров, по каждому сеансу копится история цен, фронтенд — серверные шаблоны с графиком. Фоновый сбор, Telegram-уведомления и Docker реализованы как дополнительные модули — про них в конце одной фразой».

**1:30–4:30 — Демонстрация (ядро, самое важное).**
- Афиша: карточки фильмов, поиск, фильтр «только в оригинале», отдельная страница «Новинки». [показать `/`, ввести запрос, галочку, `/new/`]
- Страница фильма: сравнение цен по кинотеатрам; график истории цены — «каждая линия это кинотеатр, нижняя где дешевле»; подсказка «когда покупать»; кнопка «Билеты» ведёт на покупку; демо-строки помечены. [показать `/movie/..`, навести на график]
- Админка: кинотеатры с галочкой «мониторим» и журнал прогонов сбора. [показать ScrapeRun]

**4:30–5:30 — Бэкенд.** «Четыре модели: кинотеатр, фильм, сеанс, снимок цены. Источники подключаются как адаптеры — новый сайт это один класс-парсер. У сеанса есть поле `source` (afisha / moskino / demo), поэтому демо и реальные данные не смешиваются. Сбор реальных данных — одна команда `scrape`».

**5:30–6:30 — Качество.** «30 автоматических тестов, парсеры проверяются на сохранённых копиях реальных страниц, линтер чистый. Каждый прогон сбора пишется в журнал со статусом — видно, что и когда собралось или почему упало». [показать журнал]

**6:30–7:30 — Ограничения и развитие.** «Внешние сайты иногда рвут соединение — поэтому в парсере есть повтор и журнал прогонов; JS парсер не исполняет. Дальше: больше кинотеатров, фоновый планировщик на Celery, уведомления в Telegram, прогноз цены — эти модули уже заложены».

**7:30–8:00 — Итог.** «Вотчер — Django-сервис мониторинга цен кино: история цен, сравнение по кинотеатрам, показы в оригинале. Спасибо за внимание».

## Вероятные вопросы (с MVP-акцентом)

- **Зачем Celery/Redis?** — «Для автосбора по расписанию. Для защиты достаточно ручной команды `scrape`; Celery — опциональный модуль и план развития».
- **Почему локально SQLite, а не Postgres?** — «Для разработки и показа хватает SQLite; PostgreSQL подготовлен для развёртывания в Docker».
- **Данные реально собираются?** — открыть журнал ScrapeRun: «да, вот прогоны со статусами; основной источник — Афиша, Москино как дополнительный».
- **Демо и реальные не путаются?** — «У сеанса есть поле `source`, демо помечено бейджем; `run.bat` демо не подсевает».
- **Что если сайт поменяет вёрстку?** — «Разбор идёт по тексту, не по CSS; тест на фикстуре сломается первым, а пустой сбор помечается ошибкой в журнале».
- **Почему не официальное API источников?** — «Публичного API с ценами нет, поэтому разбираем серверный HTML».
