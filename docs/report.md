# Введение

Поход в кино в Москве за последние годы заметно подорожал, а цена на один и тот же сеанс отличается между площадками и меняется во времени: билет на утренний показ дешевле вечернего, цена на популярный фильм растёт по мере приближения сеанса, а у репертуарных и нишевых кинотеатров действует собственная ценовая политика. При этом ни один из массовых агрегаторов (Афиша.ру, Кинопоиск, Яндекс) не хранит историю цен и не уведомляет зрителя о снижении стоимости конкретного сеанса. Зритель видит только текущую цену «здесь и сейчас» и не может ответить на практический вопрос — выгоднее купить билет сейчас или подождать.

Отдельная сложность — показы фильмов на языке оригинала с субтитрами. Такие сеансы разбросаны по нескольким площадкам (репертуарный кинотеатр «Иллюзион», городская сеть «Москино», отдельные залы сетевых кинотеатров), не имеют единой витрины и теряются в общем расписании.

**Актуальность работы** обусловлена отсутствием инструмента, который собирал бы расписание и цены кинотеатров в единую базу, отслеживал динамику цен во времени и помогал зрителю выбрать выгодный сеанс, в том числе показ в оригинале.

**Цель работы** — спроектировать и реализовать веб-сервис мониторинга цен и расписания сеансов в кинотеатрах Москвы с историей цен, уведомлениями о снижении стоимости и выделением показов на языке оригинала.

Для достижения цели поставлены следующие **задачи**:

1. проанализировать предметную область, существующие аналоги и источники данных;
2. сформировать функциональные и нефункциональные требования к сервису;
3. обосновать выбор технологического стека и спроектировать архитектуру;
4. разработать модель данных и подсистему сбора данных (парсеры) с устойчивостью к изменениям сайтов-источников;
5. реализовать веб-интерфейс афиши, личный кабинет, подписки и уведомления;
6. реализовать аналитику динамики цен и REST API;
7. покрыть систему автоматическими тестами и подготовить развёртывание.

**Объект исследования** — процессы ценообразования и публикации расписания сеансов в кинотеатрах. **Предмет** — методы автоматического сбора, хранения и анализа цен на киносеансы.

**Практическая значимость**: сервис даёт зрителю историю цен и совет «когда покупать», единую витрину показов в оригинале и персональные уведомления о снижении цены ниже заданного порога, а реализованный механизм адаптеров-парсеров позволяет подключать новые источники без изменения ядра системы.

# 1 Анализ предметной области и постановка задачи

## 1.1 Описание предметной области

Кинотеатр публикует расписание сеансов на горизонт нескольких дней. Каждый сеанс характеризуется фильмом, датой и временем начала, форматом показа (2D, 3D, IMAX), залом, признаком языка оригинала и ценой билета. Цена не статична: она зависит от времени суток, дня недели, заполняемости зала и динамически меняется по мере приближения сеанса. Таким образом, корректной единицей наблюдения является не «фильм», а «сеанс», а ценовые данные образуют временной ряд — последовательность снимков цены, снятых в разные моменты.

Источники данных делятся на два типа. Первый — сайты самих кинотеатров и городских сетей (например, расписание сети «Москино» с точными ценами). Второй — агрегаторы-афиши, публикующие сводное расписание и цену «от» для множества площадок (Афиша.ру). Оба типа отдают расписание в виде HTML-страниц, отрисованных на сервере, что делает возможным их разбор без исполнения JavaScript.

## 1.2 Обзор аналогов

Рассмотрены массовые сервисы, отображающие расписание кинотеатров Москвы.

| Сервис | Расписание и цены | История цен | Показы в оригинале | Уведомления о цене |
|---|---|---|---|---|
| Афиша.ру | Да (цена «от») | Нет | Не выделены | Нет |
| Кинопоиск | Да | Нет | Не выделены | Нет |
| Яндекс Афиша | Да | Нет | Не выделены | Нет |
| Сайты кинотеатров | Да (точная цена) | Нет | Частично | Нет |

Ни один из аналогов не сохраняет историю цен, не строит динамику и не уведомляет о снижении стоимости конкретного сеанса. Выделенной витрины показов на языке оригинала с агрегацией по нескольким площадкам также нет. Эти пробелы и определяют нишу разрабатываемого сервиса.

## 1.3 Функциональные требования

Сервис должен обеспечивать:

- автоматический периодический сбор расписания и цен из нескольких источников;
- хранение истории цен (снимок при каждом изменении) и построение графика динамики;
- витрину-афишу с фильтрами по названию и по признаку показа на языке оригинала;
- страницу фильма со сравнением цен по кинотеатрам и историей цен;
- регистрацию пользователя и подписку на фильм и/или кинотеатр с порогом цены;
- уведомления в Telegram при снижении цены ниже порога с защитой от повторов;
- аналитику динамики цены фильма и подсказку «выгоднее купить сейчас / можно подождать»;
- административную панель для контроля источников и журнала запусков парсеров;
- программный интерфейс (REST API) для доступа к данным.

## 1.4 Нефункциональные требования

- **Расширяемость**: добавление нового источника данных не должно требовать изменения ядра системы.
- **Устойчивость**: сбой одного источника не должен влиять на остальные; разбор не должен опираться на нестабильные CSS-классы вёрстки.
- **Наблюдаемость**: каждый запуск сбора фиксируется в журнале со статусом, числом сеансов и текстом ошибки.
- **Воспроизводимость**: окружение разворачивается одной командой (контейнеризация).
- **Сопровождаемость**: код покрыт автоматическими тестами и проверяется линтером.

# 2 Проектирование системы

## 2.1 Выбор технологического стека

В качестве основы выбран язык **Python 3.12** и фреймворк **Django 5.2** — зрелое решение для веб-приложений с готовой ORM, миграциями и административной панелью «из коробки», что сокращает объём типового кода. Для REST API применён **Django REST Framework**. Периодический сбор данных вынесен в фоновые задачи на **Celery 5** с брокером **Redis**, планирование — через **Celery beat**. Хранилище — **PostgreSQL 16** (в режиме разработки допускается SQLite). 

Подсистема сбора построена на **httpx** (HTTP-клиент) и **BeautifulSoup4** с парсером **lxml**. Интерфейс реализован на серверных шаблонах Django с динамическими фрагментами на **HTMX** (фильтрация без перезагрузки страницы) и графиком истории цен на **Chart.js**; вёрстка — **Bootstrap 5**. Статика отдаётся через **WhiteNoise**, прикладной сервер — **Gunicorn**. Уведомления отправляются через **Telegram Bot API**. Окружение упаковано в **Docker** и **docker-compose**. Контроль качества — **pytest** с **pytest-django** и линтер **ruff**.

Выбор обоснован зрелостью, распространённостью и взаимной совместимостью компонентов, а также соответствием теме программы профпереподготовки «Фронтенд и бэкенд разработка»: проект сочетает серверную бизнес-логику (бэкенд) и интерактивный интерфейс (фронтенд).

## 2.2 Архитектура системы

Система состоит из пяти взаимодействующих компонентов, разворачиваемых как отдельные контейнеры:

- **web** — веб-приложение Django (афиша, личный кабинет, админка, REST API);
- **worker** — исполнитель фоновых задач Celery (сбор данных, рассылка уведомлений);
- **beat** — планировщик периодических задач;
- **db** — база данных PostgreSQL;
- **redis** — брокер сообщений и хранилище результатов задач.

Планировщик `beat` по расписанию ставит задачи в очередь, `worker` их исполняет, результаты сбора сохраняются в `db`, веб-приложение читает данные из `db` и отдаёт пользователю. Такое разделение позволяет независимо масштабировать сбор данных и обслуживание запросов и обеспечивает отказоустойчивость: падение задачи сбора не влияет на доступность сайта.

## 2.3 Паттерн «парсеры-адаптеры»

Ключевое архитектурное решение — единый контракт для всех источников данных. Введён абстрактный класс `BaseParser` с единственным методом `fetch_sessions()`, возвращающим список объектов переноса данных `SessionDTO` — плоских нормализованных структур «кинотеатр, фильм, время, формат, признак оригинала, цена». Каждый источник реализуется как отдельный класс-наследник.

Наследники регистрируются в реестре автоматически через механизм `__init_subclass__`: при объявлении подкласса с заполненным `source_slug` он добавляется в общий словарь парсеров. Благодаря этому добавление нового источника сводится к созданию одного класса и одной строки импорта — ядро системы (планировщик, пайплайн сохранения, журнал) изменять не требуется. Это прямо удовлетворяет нефункциональное требование расширяемости.

## 2.4 Модель данных

Модель данных спроектирована вокруг сущности «сеанс» и истории цен. Основные сущности:

- **Cinema (кинотеатр)** — код (slug), название, сеть, город, адрес, сайт, признак нишевого кинотеатра, слаг на Афише и флаг «мониторим». Список опрашиваемых площадок хранится в базе, а не в коде: включение мониторинга — установка флага в админке.
- **Movie (фильм)** — название, оригинальное название, год, нормализованное название (для сопоставления между источниками) и признак спецпоказа. Уникальность фильма обеспечивается ограничением по паре «нормализованное название + год».
- **Session (сеанс)** — связи на кинотеатр и фильм, время начала, формат, признак языка оригинала, зал. Уникальность сеанса — по набору «кинотеатр + фильм + время + формат».
- **PriceSnapshot (снимок цены)** — связь на сеанс, минимальная и максимальная цена, момент снятия. Новый снимок создаётся только при изменении цены, что исключает дублирование и формирует компактный временной ряд.
- **Subscription (подписка)** — пользователь, фильм и/или кинотеатр, порог цены, признак активности. Ограничение целостности требует указания хотя бы одной цели подписки.
- **TelegramProfile** — привязка Telegram-аккаунта пользователя по уникальному коду (deep-link), хранит chat_id.
- **ScrapeRun (запуск парсера)** — источник, статус (выполняется/успех/ошибка), время старта и завершения, число обработанных сеансов, текст ошибки — журнал наблюдаемости.
- **NotificationLog (журнал уведомлений)** — отправленные сообщения для защиты от повторной рассылки.

Нормализация названий (приведение к нижнему регистру, замена «ё» на «е», удаление пунктуации) позволяет сопоставлять один фильм, по-разному записанный на разных сайтах. Хранение истории отдельными снимками, а не перезаписью текущей цены, является основой для аналитики динамики.

# 3 Реализация

## 3.1 Подсистема сбора данных

Реализованы два промышленных парсера и демонстрационный генератор данных для разработки интерфейса.

**Парсер Афиши.ру** обслуживает все кинотеатры с установленным флагом «мониторим» — этим закрываются и нишевый «Иллюзион» (билеты которого продаются через Афишу), и сетевой «КАРО 11 Октябрь». Запрашивается серверная страница «кинотеатр + дата». Разбор намеренно не привязан к CSS-классам, которые Афиша генерирует автоматически и часто меняет. Вместо этого карточка фильма находится подъёмом по дереву документа от ссылки на фильм до блока, содержащего одновременно время и цену, после чего блок разбирается как конечный автомат по текстовым токенам: токен формата («2D, SUB») задаёт формат и признак оригинала, токен «от 1 500 ₽» запоминается как цена, токен времени «21:35» порождает сеанс. Такой подход переживает смену вёрстки устойчивее, чем селекторы.

**Парсер «Москино»** разбирает единую страницу расписания городской сети (~20 кинотеатров). Каждый сеанс — ссылка с текстом вида «17:25 2D СУБ 420 P», который целиком распознаётся одним регулярным выражением; пометка «СУБ» означает показ на языке оригинала. Привязка сеанса к фильму и кинотеатру выполняется поиском ближайших предшествующих текстовых узлов (строки фильма вида «Название 2026 / 120 мин / США / 18+» и ближайшего названия площадки из словаря сети).

**Автообнаружение кинотеатров.** Еженедельная задача `discover_cinemas` читает каталог всех кинотеатров Москвы на Афише и дописывает в базу неизвестные площадки с выключенным мониторингом. Это решает проблему неполноты охвата: оператору достаточно установить флаг «мониторим» в админке, и со следующего цикла парсер начнёт собирать цены новой площадки без изменения кода.

## 3.2 Пайплайн сохранения

Общий для всех источников пайплайн `save_sessions` преобразует список `SessionDTO` в записи базы: методом get-or-create создаются (или находятся) кинотеатр по коду, фильм по нормализованному названию и году, сеанс по уникальному ключу. Снимок цены создаётся только если цена изменилась относительно последнего снимка — это исключает накопление одинаковых записей и делает историю компактной и информативной. Возвращается число обработанных сеансов, которое фиксируется в журнале запуска.

## 3.3 Планировщик и устойчивость сбора

Планировщик Celery beat периодически запускает задачу `run_all_parsers`, которая раздаёт по отдельной подзадаче на каждый источник — поэтому сбой одного сайта не затрагивает остальные. Каждый запуск фиксируется в журнале `ScrapeRun` (статус, число сеансов, текст ошибки), при ошибке предусмотрены повторные попытки.

Особое внимание уделено устойчивости к двум типовым проблемам веб-скрапинга. Во-первых, **блокировкам**: при получении кодов ответа 403, 429 или 503 парсер поднимает исключение `ParserBlocked`, прогон помечается в журнале как ошибка и уходит в повтор, а между запросами выдерживается настраиваемая пауза (`SCRAPER_DELAY_SECONDS`), чтобы не создавать ботоподобную нагрузку при опросе многих площадок. Во-вторых, **тихим отказам**: если из-за смены вёрстки разбор не нашёл ни одного сеанса, прогон с нулевым результатом также помечается как ошибка, а не как мнимый успех, — это делает деградацию сбора заметной в журнале.

## 3.4 Веб-интерфейс

Главная страница — афиша фильмов с предстоящими сеансами, представленная карточками. Реализованы поиск по названию и фильтры «только в оригинале» и «спецпоказы»; фильтрация выполняется без перезагрузки страницы средствами HTMX. На карточке отображается актуальная минимальная цена, рассчитанная по последним снимкам предстоящих сеансов (а не исторический минимум по всей базе), а также пометка «в оригинале». Отдельная страница «Новинки» показывает фильмы, отсортированные по году выпуска (свежие — сверху).

Страница фильма содержит сравнение предстоящих сеансов по кинотеатрам с текущими ценами и интерактивный график истории цен по кинотеатрам (Chart.js): каждая линия соответствует одному кинотеатру, что наглядно показывает, где сеанс дешевле, а где дороже, и как цены менялись во времени. Рядом выводится блок аналитики «когда покупать».

## 3.5 Личный кабинет и подписки

Реализованы регистрация и аутентификация пользователя средствами Django. В личном кабинете пользователь создаёт подписки на фильм и/или кинотеатр с указанием порога цены, просматривает и удаляет их. Форма подписки валидирует, что указана хотя бы одна цель (фильм или кинотеатр).

## 3.6 Уведомления в Telegram

Привязка Telegram реализована по схеме deep-link: на странице подписок пользователь нажимает «Привязать Telegram», переходит в бота по ссылке вида `t.me/<бот>?start=<код>`, и фоновая задача `poll_telegram_links`, опрашивая бота, по коду находит профиль и сохраняет chat_id, после чего подтверждает привязку сообщением. Задача `check_price_drops` находит предстоящие сеансы с ценой ниже порога подписки и рассылает уведомления; повторная отправка по тому же сеансу исключается журналом `NotificationLog`. При отсутствии токена бота система продолжает работать (уведомления просто не отправляются), что удобно при разработке.

## 3.7 Аналитика «когда покупать»

На основе накопленной истории цен реализован простой и интерпретируемый анализ динамики без применения машинного обучения. Функция `price_trend` сравнивает среднюю минимальную цену фильма по дням в окне в несколько суток и относит динамику к одной из категорий: «цена растёт — выгоднее купить сейчас», «снижается — можно подождать» либо «стабильна», с указанием процента изменения. Результат выводится на странице фильма, помогая зрителю принять решение о моменте покупки.

## 3.8 Программный интерфейс (REST API)

Средствами Django REST Framework реализован интерфейс только для чтения с ресурсами «фильмы» и «сеансы»; ответ по сеансу включает текущую минимальную цену. API пагинируется и пригоден для интеграции со сторонними приложениями.

# 6 Выявленные недостатки и направления развития

В ходе самостоятельного аудита проекта выявлены и частично устранены слабые места, что отражает инженерную зрелость работы.

**Устранено.** (1) Расчёт минимальной цены на афише переведён с исторического минимума по всем снимкам на минимум по последним снимкам предстоящих сеансов — теперь отображается актуальная цена. @dataclass
class SessionDTO:
    """Нормализованный сеанс из любого источника."""
    cinema_slug: str
    cinema_name: str
    movie_title: str
    starts_at: datetime
    price_min: Decimal | None
    price_max: Decimal | None = None
    format: str = "2D"
    original_language: bool = False


class BaseParser(ABC):
    source_slug: str = ""
    source_name: str = ""

    def __init_subclass__(cls, **kwargs):
        super().__init_subclass__(**kwargs)
        if cls.source_slug:
            _REGISTRY[cls.source_slug] = cls

    @abstractmethod
    def fetch_sessions(self) -> list[SessionDTO]:
        """Вернуть будущие сеансы источника с ценами."""

# Приложение Б. Пайплайн сохранения и аналитика

```python
def save_sessions(dtos: list[SessionDTO]) -> int:
    processed = 0
    for dto in dtos:
        cinema, _ = Cinema.objects.get_or_create(
            slug=dto.cinema_slug, defaults={"name": dto.cinema_name})
        movie, _ = Movie.objects.get_or_create(
            normalized_title=normalize_title(dto.movie_title), year=dto.movie_year,
            defaults={"title": dto.movie_title,
                      "is_special": is_special_title(dto.movie_title)})
        session, _ = Session.objects.get_or_create(
            cinema=cinema, movie=movie, starts_at=dto.starts_at, format=dto.format)
        processed += 1
        last = session.prices.order_by("-collected_at").first()
        if last and last.price_min == dto.price_min:
            continue  # цена не изменилась — не плодим одинаковые снимки
        PriceSnapshot.objects.create(session=session, price_min=dto.price_min)
    return processed
```

# Приложение В. Журнал запусков и устойчивость сбора

```python
@shared_task(bind=True, max_retries=2, default_retry_delay=300)
def run_parser(self, slug: str) -> int:
    run = ScrapeRun.objects.create(source=slug)
    try:
        dtos = get_parser(slug)().fetch_sessions()
        run.sessions_found = services.save_sessions(dtos)
        if run.sessions_found == 0:
            run.status = ScrapeRun.Status.ERROR
            run.error = "Получено 0 сеансов: вероятна смена вёрстки или блокировка."
        else:
            run.status = ScrapeRun.Status.SUCCESS
    except Exception as exc:
        run.status = ScrapeRun.Status.ERROR
        run.error = repr(exc)
        run.save()
        raise self.retry(exc=exc)
    run.finished_at = timezone.now()
    run.save()
    return run.sessions_found
```

Параметр `SCRAPER_DELAY_SECONDS` задаёт паузу между запросами, а коды ответа 403/429/503 поднимают исключение `ParserBlocked` — прогон помечается ошибкой и уходит в повтор, что защищает от блокировок и делает деградацию сбора заметной.
