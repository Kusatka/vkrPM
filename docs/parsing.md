# Как работает парсинг в «Вотчере»

Небольшой доклад о функциях подсистемы сбора цен (приложение `apps/scraping`).

## 1. Общий цикл

Каждые 4 часа Celery beat запускает задачу `run_all_parsers` (`tasks.py`). Она
не делает работу сама, а ставит по отдельной задаче `run_parser(slug)` на каждый
зарегистрированный источник. Это ключевое решение для надёжности: если Афиша
вернула капчу, задача Москино об этом даже не узнает.

`run_parser` создаёт запись журнала `ScrapeRun`, вызывает у парсера
`fetch_sessions()`, передаёт результат в `services.save_sessions()` и закрывает
журнал статусом success или error (с текстом исключения и двумя ретраями через
5 минут). Журнал виден в админке: «Сбор цен → Запуски парсеров» — это первый
экран, куда нужно смотреть, если данные перестали обновляться.

Запустить сбор вручную (например, наполнить базу реальными фильмами перед
защитой) можно командой `python manage.py scrape` — она опрашивает все
источники синхронно, с выводом в консоль, и так же пишет каждый прогон в
`ScrapeRun`. Можно указать один источник: `scrape moskino`. Демо-данные
(4 фильма для офлайн-показа интерфейса) кладёт отдельная команда `seed_demo` —
её не путать с боевым сбором.

## 2. Контракт парсера

Все источники реализуют один интерфейс (`parsers/base.py`):

```python
class BaseParser(ABC):
    source_slug: str
    def fetch_sessions(self) -> list[SessionDTO]: ...
```

`SessionDTO` — нормализованный сеанс: кинотеатр (слаг, название, сеть, флаг
нишевости), фильм (название, оригинальное название, год), время начала
(обязательно с таймзоной), формат, флаг «на языке оригинала», цены min/max и
ссылку на страницу покупки билета (`url`).
Парсер ничего не знает о моделях Django — он возвращает плоские DTO, поэтому
его разбор тестируется без базы данных.

Регистрация автоматическая: `BaseParser.__init_subclass__` кладёт каждый
подкласс в реестр по `source_slug`. Добавление источника — один новый класс
и одна строка импорта в `parsers/__init__.py`.

## 3. Парсер Афиши.ру (`parsers/afisha.py`)

Источник — страница «кинотеатр + дата»
(`afisha.ru/msk/cinema/<slug-id>/movie/<ДД-ММ-ГГГГ>`), она рендерится на
сервере. Список кинотеатров для опроса берётся из БД: `monitored_cinemas()`
возвращает все записи Cinema с галочкой «Мониторим» и заполненным
`afisha_slug` (если БД пуста — встроенная пара Иллюзион + КАРО 11 Октябрь).

Разбор двухступенчатый и нарочно не привязан к CSS-классам, потому что классы
у Афиши генерируются сборщиком и меняются чаще, чем структура контента:

- `_movie_block(link)` — от каждой ссылки на фильм поднимаемся вверх по DOM,
  пока не найдём контейнер, в котором есть и время сеанса, и цена (или «Нет
  билетов»). Если поднялись до общего списка, где фильмов больше одного, —
  значит карточкой был предыдущий уровень.
- `_parse_block(...)` — внутри карточки идём по текстовым токенам как конечный
  автомат: токен формата («2D», «2D, SUB», «IMAX») задаёт текущий формат и
  признак оригинала; токен «от 1 500 ₽» запоминается как цена; токен-время
  «21:35» порождает SessionDTO с накопленным состоянием и сбрасывает цену.
  «Нет билетов» даёт сеанс без цены. Год фильма ловится по токену вида
  «2026, Биография» сразу после названия.

Пометка SUB на Афише означает показ на языке оригинала с субтитрами — так
сервис узнаёт «нишевые» сеансы.

## 4. Парсер Москино (`parsers/moskino.py`)

Источник — `mos-kino.ru/schedule/?date=ГГГГ-ММ-ДД`: одна страница со всеми
~20 кинотеатрами городской сети, тоже серверный рендер. Каждый сеанс — это
ссылка `javascript:ticketManager.richSession(ID)` с текстом вида
«17:25 2D СУБ 420 P», который целиком разбирается одной регуляркой
(время, формат, признак субтитров, точная цена).

Привязка сеанса к фильму и кинотеатру — через `find_previous`: ближайшая
предшествующая строка вида «Название ГГГГ / 120 мин / Страна / 18+» — это
фильм (название и год вырезаются регулярками, кавычки-ёлочки срезаются);
ближайший предшествующий текст из словаря кинотеатров — это площадка.
Словарь встроен в код и дополняется из БД (`_cinema_map`): кинотеатр сети,
добавленный через админку с network=«Москино», подхватится без правки кода.

## 5. Сохранение: `services.save_sessions()`

Единый пайплайн для всех источников. Для каждого DTO:

1. `Cinema.get_or_create` по слагу — кинотеатры создаются при первом сеансе;
2. `Movie.get_or_create` по паре (нормализованное название, год).
   `normalize_title()` приводит к нижнему регистру, заменяет «ё», выбрасывает
   пунктуацию — поэтому «Питер FM» с Москино и «Питер ФМ» с Афиши не плодят
   дубликаты, а «Дюна» 1984 и 2021 годов остаются разными фильмами;
3. `Session.get_or_create` по ключу (кинотеатр, фильм, время, формат) —
   повторный прогон не создаёт дублей; у сеанса сохраняется ссылка на страницу
   источника — это кнопка «Билеты» в таблице «Где смотреть» на странице фильма;
4. `PriceSnapshot` пишется только если цена изменилась с прошлого снимка.
   Именно из этих снимков строится график на странице фильма — история цен
   и есть продукт сервиса.

## 6. Обнаружение новых кинотеатров (`discovery.py`)

Проблема: мы можем просто не знать о существовании площадки. Решение —
еженедельная задача `discover_cinemas`: она читает каталог кинотеатров Москвы
на Афише.ру (`/msk/cinema/cinema_list/`, все площадки города), извлекает пары
«название + afisha_slug» и дописывает в БД те, которых ещё нет, со снятой
галочкой «Мониторим». Человек заходит в админку, видит новые кинотеатры и
включает галочку у интересных — со следующего цикла парсер их опрашивает.
То же руками: `python manage.py discover_cinemas`.

## 7. Тестирование и деградация

Функции разбора — чистые: HTML на входе, DTO на выходе. Тесты гоняют их на
фикстурах — сохранённых копиях реальных страниц (сверены 10.06.2026). Если
источник сменит вёрстку, сломается тест на новой фикстуре, а в проде упадёт
конкретный ScrapeRun — остальные источники продолжат собирать. Сценарии
эксплуатации: интервал 4 часа, User-Agent с пометкой учебного проекта,
точечные запросы страниц вместо обхода сайтов.
