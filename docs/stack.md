# Технологический стек — «Вотчер»

Сервис мониторинга цен и расписания сеансов в кинотеатрах Москвы. Ниже — полный
стек с кратким обоснованием каждого выбора.

## Сводка

| Слой | Технология | Назначение |
|---|---|---|
| Язык | Python 3.12+ | основной язык проекта |
| Веб-фреймворк | Django 5.2 | модели, ORM, админка, маршрутизация, шаблоны |
| REST API | Django REST Framework 3.15 | выдача данных в JSON (`/api/`) с пагинацией |
| База данных | PostgreSQL (прод) / SQLite (локально) | хранение кинотеатров, фильмов, сеансов, истории цен |
| Очередь задач | Celery 5.4 | фоновый сбор цен и рассылка уведомлений |
| Планировщик | Celery beat | периодический запуск парсеров (каждые 4 ч) |
| Брокер/результаты | Redis | транспорт для Celery |
| HTTP-клиент | httpx | загрузка страниц источников |
| Разбор HTML | BeautifulSoup 4 + lxml | извлечение сеансов и цен из текста страниц |
| Фронтенд | Django-шаблоны + Bootstrap 5 | вёрстка и базовые компоненты |
| Интерактив | HTMX | фильтры/поиск без перезагрузки страницы |
| Графики | Chart.js | график истории цены по кинотеатрам |
| Уведомления | Telegram Bot API | оповещения о снижении цены |
| Статика | WhiteNoise | отдача статики без отдельного сервера |
| Тесты | pytest + pytest-django | 30 тестов: парсеры, представления, аналитика |
| Качество кода | ruff | линтер и сортировка импортов |
| Запуск/деплой | Docker + docker-compose, Gunicorn | веб, БД, Redis, worker и beat одной командой |

## Обоснование ключевых решений

**Django + DRF.** Готовая ORM, миграции и админка закрывают весь CRUD и
ручное управление кинотеатрами (галочка «Мониторим»), а DRF бесплатно даёт
REST API поверх тех же моделей. Это минимизирует объём кода для учебного срока.

**PostgreSQL в проде, SQLite локально.** Подключение задаётся одной строкой
`DATABASE_URL`, поэтому защита и разработка идут на SQLite без установки СУБД, а
боевой стенд — на PostgreSQL. Код приложения при этом не меняется.

**Celery + Redis + beat.** Сбор цен — длительная сетевая операция, её нельзя
держать в веб-запросе. Celery выносит её в фон, beat запускает по расписанию
(каждые 4 часа), а падение одного источника не валит остальные (отдельная
задача на источник).

**httpx + BeautifulSoup (а не Selenium).** Источники отдают готовый серверный
HTML, поэтому headless-браузер избыточен. Разбор намеренно идёт по тексту
страницы (формат → цена → время), а не по CSS-классам — так парсер переживает
смену вёрстки. Ограничение: JavaScript не исполняется (отмечено в «Известных
ограничениях»).

**HTMX + Chart.js вместо SPA.** Нужны точечная интерактивность (фильтры, поиск)
и один график. Полноценный фронтенд-фреймворк здесь — лишняя сложность; HTMX
обновляет фрагменты страницы, Chart.js рисует историю цен по кинотеатрам.

**pytest + ruff + Docker.** Тесты парсеров на сохранённых фикстурах не зависят
от живых сайтов; ruff держит стиль; docker-compose поднимает весь стек (web, db,
redis, worker, beat) одной командой для воспроизводимости на защите.

## Структура проекта

```
kinomonitor/
├── apps/
│   ├── catalog/        модели (Cinema, Movie, Session, PriceSnapshot), views, API, графики
│   ├── scraping/       парсеры (afisha, moskino, demo), пайплайн сохранения, задачи Celery,
│   │                   команды: scrape (реальные данные), seed_demo (демо), discover_cinemas
│   ├── accounts/       подписки и привязка Telegram
│   └── notifications/  проверка снижения цены и отправка уведомлений
├── config/             settings, urls, celery
├── templates/          base + страницы каталога и аккаунта
└── tests/              pytest + фикстуры HTML источников
```
